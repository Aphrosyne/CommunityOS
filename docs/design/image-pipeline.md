# Image Pipeline 图片处理流水线

> **状态：** 正式
> **版本：** v1.0
> **最后更新：** 2026-07-01

---

# 目的

本文档定义 CommunityOS 图片处理流水线的整体设计。

图片处理流水线是一个独立于聊天平台的图片处理模块，负责处理社区内的图片投稿、预处理、发布以及后续扩展功能。

该模块应尽可能保持平台无关，可被 QQ、Discord 或未来其他平台共同调用。

---

# 设计目标

图片处理流水线的设计目标包括：

* 降低交流群直接传播高风险图片的风险。
* 将图片投稿流程标准化。
* 减少管理员处理图片内容的重复劳动。
* 为未来跨平台同步提供统一接口。
* 保持模块独立，便于维护和扩展。

---

# 模块职责

图片处理流水线负责：

* 接收图片投稿。
* 图片预处理。
* 图片混淆。
* 图片发布。
* 图片缓存管理。
* 图片日志记录。
* 后续扩展接口。

图片处理流水线不负责：

* 图片内容审核。
* 社区处罚。
* 用户权限管理。
* 平台消息收发。

# 已实现功能（v1.0）

## 发布 (publish / 发布)

私聊发送「发布」→ 进入发布模式 → 发送图片 → 发送「完成」开始发布。

工作流程：

```text
用户

↓

私聊发送「发布」

↓

进入发布模式

↓

发送图片（最多 10 张）

↓

发送「完成」

↓

Gilbert 曲线混淆

↓

合并为一条消息发送到指定群
（@投稿人 + 混淆图 + 解混淆网址）

↓

清理临时缓存

↓

结束
```

整个流程中，交流群不直接接收用户原始图片。

配置参数：上限 10 张，会话超时 3 分钟，动态冷却 `min(30 + 10×张数, 90)` 秒。

## 混淆 (obfuscate / 混淆)

私聊发送「混淆」→ 进入混淆模式 → 发送图片 → 发送「完成」开始混淆。

与发布流程相同，区别在于混淆图由私聊一条消息返回，不发群。

共用发布插件的全部上限、超时和冷却参数。

## 解图 (decode / 解图)

三种触发方式：

**方式一：会话模式** — 私聊「解图」→ 进入解混淆模式 → 发送图片 → 发送「完成」→ 返回原图。与混淆插件结构一致，方向相反。

**方式二：自动识别** — 私聊中直接转发群里的混淆消息（消息文本含 `IMAGE_DECODE_URL`），机器人自动识别并即时解混淆返回。无需进入会话，无需指令。

**方式三：群聊引用** — 群聊中引用一条含图消息，发送 `@bot 解图`，机器人提取被引用消息中的图片，解混淆后私信发送给操作用户。群内无回复。

三种方式共用动态冷却公式和冷却池。

## 基础设施

图片三种功能共用以下公共服务：

- **SessionService**（`services/session.py`）— 管理多步交互会话：创建、完成、取消、超时扫描及通知。
- **ThrottleService**（`services/throttle.py`）— 按 `(user_id, reply_type)` 控制机器人回复频率，防刷屏。
- **动态冷却**：`cooldown = min(base + per_image × count, max)`，默认 `min(30 + 10×N, 90)` 秒。
- **asyncio.Lock** — 每用户串行化 session 操作，防止并发竞态。

# 图片混淆

混淆属于图片处理流程的核心环节。

当前采用 **Gilbert 曲线 + 黄金比例偏移** 像素置换算法（小番茄）。

由 `services/image_obfuscator.py` 独立实现，提供 `obfuscate()`（ENC）和 `deobfuscate()`（DEC）两个函数，numpy 加速。

混淆图为 JPEG 格式，信息完整保留，可通过在线解混淆网址或解图插件还原。

发布时随图附带在线解混淆网址，供群成员还原查看。

---

# 图片发布

图片处理完成后，机器人负责：

* 上传混淆后的图片。
* @ 投稿用户。
* 附带解混淆网址。
* 发送到 `.env` 配置的指定交流群。

发布行为应统一由平台适配层完成。

图片处理流水线不直接调用平台 API。

# 配置项

已实现的配置项：

```ini
IMAGE_SUBMIT_GROUP=0            # 发布目标群号
IMAGE_DECODE_URL=...            # 在线解混淆网址
PUBLISH_MAX_IMAGES=10           # 单次提交图片上限
PUBLISH_TIMEOUT=180             # 会话超时（秒）
PUBLISH_COOLDOWN_BASE=30        # 冷却基数（秒）
PUBLISH_COOLDOWN_PER_IMAGE=10   # 每张图片追加冷却（秒）
PUBLISH_COOLDOWN_MAX=90         # 冷却上限（秒）
SESSION_SCAN_INTERVAL=10        # 超时扫描间隔（秒）
REPLY_THROTTLE=5                # 回复节流窗口（秒）
```

后续计划增加：

```text
投稿群列表（多群选择）
最大图片尺寸
允许投稿用户白名单
```

所有配置均采用外部配置文件。

# 日志记录

图片处理流程应记录必要日志。

建议记录：

* 时间
* 投稿用户
* 图片尺寸
* 处理状态
* 是否成功
* 异常信息

日志用于排查问题、调试程序和后续统计。

日志不保存图片内容。

---

# 缓存管理

处理过程中允许使用临时缓存。

处理完成后应及时删除原图缓存和中间文件，避免长期保留无意义数据。

发送失败或处理异常时，临时文件由 `_cleanup_tmp()` 统一清理。

---

# 错误处理

当图片处理失败时，机器人应：

* 返回失败提示。
* 记录错误日志。
* 不影响其他图片继续处理。

异常不得导致整个机器人退出。

---

# 插件关系

图片处理流水线不直接作为机器人，而是由图片插件调用。

关系如下：

```text
图片插件（publish / obfuscate / decode）

↓

图片处理流水线（image_obfuscator.py）

↓

平台适配层
```

图片处理流水线属于公共能力，三个图片插件共同调用。

# 总结

图片处理流水线是 CommunityOS 的独立图片处理模块。

它负责统一图片投稿流程、执行图片混淆处理、提供标准化接口，并为未来的跨平台扩展保留能力。

其核心目标不是增加功能，而是在保证模块独立性的前提下，降低社区图片管理成本，提高社区长期运行的稳定性。
